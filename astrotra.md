Most astrology products today fall into one of these buckets:

1\. generic daily horoscope apps,  
2\. astrologer marketplace apps,  
3\. chart-generator sites with poor UX,  
4\. generic AI chatbots with astrology-themed language.

This product should create a new category:

\#\# Category Definition  
A \*\*chart-aware, multi-method astrology AI search engine\*\*.

Instead of choosing an LLM model like GPT / Gemini / Claude, the user chooses an \*\*astrology method\*\* such as:

\- Vedic  
\- KP  
\- Western  
\- Compare All

The key insight is:

So the product must feel like:

\*\*a personal guidance engine powered by structured astrological interpretation.\*\*

\# 19\. Core Technical Architecture

Design the product in layers.

\#\# Layer 1 — Astronomical Calculation Layer  
This must be deterministic, precise, and non-LLM.

Use:  
\- Swiss Ephemeris / pyswisseph as source of truth

This layer computes:  
\- planetary positions,  
\- signs,  
\- degrees,  
\- nodes,  
\- retrograde status,  
\- house cusps,  
\- ascendant,  
\- midheaven,  
\- timing-related astronomical inputs.

\#\# Layer 2 — Chart Normalization Layer  
Create a canonical chart object that all methods can read.

This object should include:  
\- planets  
\- signs  
\- houses  
\- degrees  
\- retrograde  
\- aspects  
\- house rulers / lords  
\- ascendant  
\- moon sign  
\- nakshatras where relevant  
\- divisional references where relevant  
\- timing metadata  
\- confidence metadata

\#\# Layer 3 — Method Adapter Layer  
Build separate logic engines:

\- Vedic Engine  
\- KP Engine  
\- Western Engine

Each engine should:  
\- read from the canonical chart object,  
\- apply its own method-specific logic,  
\- output structured evidence,  
\- never depend on vague text generation alone.

\#\# Layer 4 — Query Analysis Layer  
For every user question:  
\- classify it,  
\- determine analysis depth,  
\- estimate timing needs,  
\- estimate birth-time sensitivity,  
\- decide which engine(s) to invoke,  
\- extract only relevant evidence.

\#\# Layer 5 — Evidence Extraction Layer  
Before answer generation, build a structured evidence object.

\#\# Layer 6 — Answer Composition Layer  
Use the LLM only to turn structured evidence into elegant, readable responses.

\#\# Layer 7 — Memory / Personalization Layer  
Store:  
\- user preferences,  
\- repeated themes,  
\- prior questions,  
\- favored method,  
\- favored tone,  
\- resonance feedback,  
\- saved readings,  
\- linked charts.

\# 20\. Method Engine Requirements

\#\# 20.2 KP Engine  
Treat KP as a precision/timing-focused interpretive engine.

Use it especially for:  
\- “when” questions,  
\- period favorability,  
\- event timing,  
\- narrow decision windows,  
\- query-specific timing judgment.

Its outputs should feel more exacting and timing-sensitive than generic Vedic output.

\#\# 20.3 Western Engine  
Design this as a tropical-framework analysis engine.

It should support:  
\- natal interpretation,  
\- aspects,  
\- houses,  
\- personality patterns,  
\- emotional dynamics,  
\- relationship style,  
\- current transit framing,  
\- self-discovery questions.

\#\# 20.4 Compare Mode  
When compare mode is selected:  
\- run engines independently,  
\- do not merge logic before outputs are computed,  
\- synthesize only at the end.

The comparison layer should return:  
\- strongest shared themes,  
\- strongest differences,  
\- practical robustness notes,  
\- which conclusions seem strongest across methods.

\# 24\. Confidence Framework

Every answer must carry internal confidence metadata.

\#\# Confidence Components  
\- input confidence  
\- birth-time confidence  
\- chart confidence  
\- method confidence  
\- timing confidence  
\- evidence strength  
\- cross-method agreement  
\- final answer confidence

\#\# Logic  
\#\#\# If birth time is unknown  
\- reduce confidence in house-sensitive and timing-sensitive claims.

\#\#\# If multiple methods agree  
\- increase robustness.

\#\#\# If conclusion depends on narrow evidence  
\- mark as tentative.

\#\#\# If question is heavily timing-based and user lacks birth time  
\- down-rank prediction specificity.

The user-facing output should express confidence in natural language, not raw scores.
